# 기본 요구사항 정리

## 구현 범위
- 회원 등급: `NORMAL`, `VIP`, `VVIP`
- 주문: 상품명, 주문 원가, 회원 정보 저장
- 결제 수단: `CREDIT_CARD`, `POINT`
- 결제 성공 시 주문 정보, 할인 금액, 최종 결제 금액, 결제 수단, 결제 시각 저장
- 회원 등급별 할인 정책 테스트 작성

## 설계에서 중요하게 본 점
- 할인 정책은 인터페이스 기반으로 분리해 정책 추가/변경에 유연하게 대응할 수 있도록 구성했습니다.
- 주문과 결제 엔티티를 분리해 주문 상태와 결제 이력을 명확히 구분했습니다.
- 서비스는 흐름 제어에 집중하고, 계산 규칙과 생성 로직은 별도 객체로 분리했습니다.

## 결제 멱등성에 대한 고민
- 같은 주문에 대한 중복 결제를 막기 위해 주문 상태를 `READY -> PAID`로 조건부 변경했습니다.
- 결제 테이블의 `order_id`는 unique 제약으로 한 번만 저장되도록 설계했습니다.
- 이미 결제된 주문으로 다시 요청이 오면 중복 저장 대신 기존 결제 결과를 재반환하도록 처리했습니다.

## 동시성에 대한 고민
- 단순 조회 후 저장만 하는 방식은 동시 요청에서 취약하다고 판단했습니다.
- 주문 상태를 조건부 업데이트하는 방식으로 한 순간에 하나의 요청만 결제 성공하도록 제어했습니다.
- 최종 방어선은 DB unique 제약으로 두어 애플리케이션 레벨 체크에만 의존하지 않도록 구성했습니다.

## 할인 정책 적용 방식에 대한 고민
- 추가 요구사항을 감안해 적용 가능한 여러 정책을 모두 계산할 수 있는 구조로 설계했습니다.
- 정책 목록은 우선순위 기준으로 정렬해 적용 순서가 코드에 명시되도록 구성했습니다.
- 여러 정책 합산 결과가 주문 금액을 넘지 않도록 과할인을 방지했습니다.

## 테스트
- `NORMAL`: 할인 없음 검증
- `VIP`: 1,000원 고정 할인 검증
- `VVIP`: 10% 할인 및 절사 규칙 검증
- 결제 성공, 중복 요청, 존재하지 않는 주문 결제에 대한 통합 테스트 추가
