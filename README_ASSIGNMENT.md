# 기본 요구사항 정리

## 구현 범위
- 회원 등급: `NORMAL`, `VIP`, `VVIP`
- 주문: 상품명, 주문 원가, 회원 정보 저장
- 결제 수단: `CREDIT_CARD`, `POINT`
- 결제 성공 시 주문 정보, 할인 금액, 최종 결제 금액, 결제 수단, 결제 시각 저장
- 회원 등급별 할인 정책 테스트 작성

## 설계에서 중요하게 본 점
- 할인 정책은 인터페이스 기반으로 분리해 정책 추가/변경에 유연하게 대응할 수 있도록 구성
- 주문과 결제 엔티티를 분리해 주문 상태와 결제 이력을 명확히 구분
- 서비스는 흐름 제어에 집중하고, 계산 규칙과 생성 로직은 별도 객체로 분리

## 결제 멱등성에 대한 고민
- 같은 주문에 대한 중복 결제를 막기 위해 주문 상태를 `READY -> PAID`로 조건부 변경
- 결제 테이블의 `order_id`는 unique 제약으로 한 번만 저장되도록 설계했습니다.
- 이미 결제된 주문으로 다시 요청이 오면 중복 저장 대신 기존 결제 결과를 재반환하도록 처리

## 동시성에 대한 고민
- 단순 조회 후 저장만 하는 방식은 동시 요청에서 취약하다고 판단
- 주문 상태를 조건부 업데이트하는 방식으로 한 순간에 하나의 요청만 결제 성공하도록 제어
- 최종 방어선은 DB unique 제약으로 두어 애플리케이션 레벨 체크에만 의존하지 않도록 구성

## 할인 정책 적용 방식에 대한 고민
- 추가 요구사항을 감안해 적용 가능한 여러 정책을 모두 계산할 수 있는 구조로 설계
- 정책 목록은 우선순위 기준으로 정렬해 적용 순서가 코드에 명시되도록 구성
- 여러 정책 합산 결과가 주문 금액을 넘지 않도록 과할인을 방지

## 테스트
- `NORMAL`: 할인 없음 검증
- `VIP`: 1,000원 고정 할인 검증
- `VVIP`: 10% 할인 및 절사 규칙 검증
- 결제 성공, 중복 요청, 존재하지 않는 주문 결제에 대한 통합 테스트 추가

# 추가 요구사항 정리

## 구현 범위
- 결제 시 적용된 할인 정책 정보를 `DiscountHistory`로 별도 저장
- 포인트 결제 시 등급 할인 후 금액 기준으로 추가 5% 할인 적용
- 할인 정책명, 할인율, 할인 금액, 적용 당시 회원 등급을 스냅샷으로 보존

## 설계에서 중요하게 본 점
- 결제 엔티티에는 총 할인 금액만 저장하고, 정책별 상세 내역은 별도 이력 테이블로 분리
- 등급 할인과 결제 수단 할인을 분리해 중복 할인 규칙을 명확히 표현
- 할인 이력은 정책 객체 참조가 아니라 값 스냅샷으로 저장해 정책 변경과 분리

## 데이터 정합성에 대한 고민
- 최종 결제 금액에 반영된 할인 총액과 할인 이력 합계가 항상 일치하도록 할인 결과를 정규화
- 등급 할인과 결제 수단 할인은 각각 적용 가능한 최대 금액을 넘지 않도록 제한
- 결제 수단 할인은 원가가 아니라 등급 할인 적용 후 금액을 기준으로 계산

## 테스트
- VIP/VVIP + 포인트 결제 시 중복 할인 순서와 최종 금액 검증
- 할인 이력 저장 여부 및 할인 타입별 저장 값 검증
- 할인 이력 합계와 실제 결제 할인 총액 일치 여부 검증
